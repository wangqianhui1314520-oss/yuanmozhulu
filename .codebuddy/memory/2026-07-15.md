## 前端 SPA 路由 404 彻底修复 (2026-07-15)

问题：`/game`、`/faction` 等前端 SPA 路由返回 404，游戏无法游玩。

根因分析：
1. `app.mount("/", StaticFiles(html=True))` 在模块级别会拦截所有请求，使之后定义的 catch-all route 失效
2. FastAPI 的 Mount 优先级机制：所有 mount 的请求处理在 route 匹配之前
3. `StaticFiles(html=True)` 的 SPA fallback 在 Starlette 1.3.1 中未生效

最终方案：移除所有 `app.mount("/", ...)`，改用智能 catch-all route：
- `@app.get("/{path:path}")` 放在所有 API 路由之后
- 若请求路径对应 `dist/` 中真实文件 → `FileResponse` 直接返回（含正确 MIME）
- 否则 → 返回 `index.html`（SPA fallback）
- 含路径穿越安全校验

关键教训：FastAPI/Starlette 中 `app.mount("/")` 不应与 catch-all route 混用，mount 会劫持所有路径。直接从 route 中手动分发是最可靠的方案。

## 面板交互"假交互"修复 (2026-07-15)

### 问题分析
上轮为7个面板新增的交互功能中，9个函数仅写入前端 `store.addEvent()` 本地事件日志，没有调用后端 API，对游戏世界状态**零影响**（银两未扣、声望未加、朝纲未变）。

### 修复方案
每个函数增加两步：**`API.submitCommand()` 提交到后端指令队列** + **即时修改 Pinia 本地状态**（确保 UI 实时响应）。

原有"真交互"函数使用的 action 名称（`relieve_disaster`/`promote_official`/`demote_official`/`build_ship`/`upgrade_building`/`open_trade_route`/`treat_injured`）不在后端 `valid_actions` 白名单中，全部映射为合法 action：

| 原 action | 映射为 | type 标识 |

| `relieve_disaster` | `relief` | `type: 'relieve'` |
| `promote_official` | `appoint` | `type: 'promote'` |
| `demote_official` | `dismiss` | `type: 'demote'` |
| `build_ship` | `build` | `building: 'ship'` |
| `upgrade_building` | `build` | `building: 'shipyard'/'clinic'` |
| `open_trade_route` | `trade` | `type: 'open_route'` |
| `treat_injured` | `relief` | `type: 'treat_injured'` |

### 修复的9个"假交互"函数详情

| 函数 | 后端 action | 即时本地状态修改 |
